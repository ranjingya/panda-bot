# AGENTS.md

本文件适用于仓库根目录及全部子目录，用于约束参与本项目的 AI 代理。执行任务时，以用户当前指令为最高优先级；本文件中的已确认规则高于 `docs/README.md` 中仍标注为“建议”的默认参数。

## 沟通与执行

1. 每轮先把用户的自然语言需求转换为“本轮执行提示词”，并在回复第一行展示。
2. 回答先给结论，再给必要过程。
3. 能在当前授权范围内完成的任务直接完成，不只提供建议。
4. 不确定的信息必须明确标注，并通过代码、测试、配置或官方资料验证，禁止编造。
5. 产品规则存在歧义时，优先提出会影响用户体验、隐私或架构的关键问题。
6. 文档只描述当前最终状态，不记录讨论过程、调试过程或版本对比。

## 项目目标

“盼达 Panda”是飞书群聊中的事件驱动型下班氛围机器人。它优先根据团队近期出现的真实工作收尾信号判断，并以低优先级时间规则兜底，在不可预测的时刻偶尔发言，为群成员制造接近下班的期待感。

项目必须遵守以下原则：

- 不使用固定时间播报或机械倒计时。
- 不催任务，不评价个人，不承担考勤或绩效统计职责。
- 普通消息只触发计算，消息数量不直接转化为自由能量。
- 时间仅作为普通消息未命中收尾信号后的低优先级兜底条件，不覆盖消息分类结果。
- 明确规则负责频率、门槛、概率、冷却和上限，运行时 AI 不拥有最终触发权。
- 默认不长期保存群聊正文，不建立个人画像或个人排行榜。

## 已确认的产品规则

### 工作时间与触发时间

- 时区固定为 `Asia/Shanghai`。
- 上午工作时段为 `08:35–11:30`。
- 下午工作时段为 `12:45–17:30`。
- 上午的有效收尾信号会积累能量，并经过午休和时间间隔衰减后影响下午。
- `12:45` 起继续积累下午能量。
- `14:30` 是最早允许发言的时间边界，不是定时触发点。
- 机器人只在新事件到达时进行判断；没有新事件时不得主动唤醒。
- `14:30` 后的有效普通对话轮次可按外部配置的时间概率参与兜底抽签，概率随时间区间变化。
- `17:30` 后绝对静默。发送消息前必须再次检查当前时间，互动回复也遵守此限制。

### 活跃度与休息日

- 周末和节假日通过群聊实际活跃度判断，不依赖消息数量增加能量。
- 活跃度统计只包含真实成员产生的有效对话轮次。
- 同一成员在 60 秒内连续发送的短消息合并为一个对话轮次。
- 机器人消息、系统消息、图片、文件、贴纸、转发内容和重复事件不计入活跃度。
- 下午至少两名真实成员参与聊天，当前日期才具备触发资格。
- 活跃度门槛必须位于外部配置中，并通过影子模式数据校准。

### 自由能量与有效信号

- 只有真实工作收尾信号增加自由能量，例如完成、跑通、改好、发布、交付、待交付、基本稳定和阶段性结束。
- 单独出现“OK”“好了”“终于”等含义不完整的词时，必须结合短期上下文判断。
- “下班、想走、想休息”等表达不是能量信号，也不提供召唤能力。
- “还没完成、完成不了、仍在报错、准备返工”等表达不得识别为收尾。
- “烦死了、服了、要命、崩溃”等日常吐槽不增加能量，也不触发沉默。
- 明确的线上故障、数据异常、客户投诉、紧急修复、回滚或业务影响会触发临时沉默。
- 风险抑制时长、能量分数、衰减曲线、隐藏门槛和抽签概率必须由外部配置提供。
- 概率不得达到 100%，达到门槛后仍允许抽签失败。
- 时间兜底不增加、消耗或重置自由能量；未完成表达、忽略表达、日常吐槽和风险消息不参与时间兜底。

### 群体条件、频率与冷却

- 测试群由 3 名真实成员组成，日常聊天频繁且存在大量拆句。
- 至少两名成员参与下午聊天即可满足群体活跃条件。
- 只需一名成员贡献明确收尾信号，便可以推动最终触发。
- 同一成员短时间重复发送相似完成表达时只计算一次。
- 每日主动发言最多 4 次，达到上限后保持静默。
- 时间兜底每天最多触发 1 次，计入每日主动发言总上限，并始终使用独立群消息。
- 任意两次主动发言之间的随机冷却时间为 30～45 分钟。
- 收尾信号触发后保留少量能量，并提高下一次收尾触发门槛；时间兜底不改变能量和门槛。
- 每日上限表示最大可能次数，不保证机器人当天一定出现。

### 发言与互动形式

- 主动发言可在独立群消息和回复近期明确收尾消息之间随机切换。
- 初始建议权重为独立群消息约 60%、回复消息约 40%，具体权重放入配置。
- 回复消息时不得评价个人，不得说明具体识别了谁的工作量。
- 含敏感内容、长文本、转发内容或含义模糊时使用独立群消息。
- 每次主动发言后开启 5 分钟互动窗口。
- 互动窗口内收到回复或 `@盼达` 时，最多随机回嘴一次。
- 回嘴不增加能量，不重置主动发言冷却，也不计为一次主动发言。
- `17:30` 后不得发送回嘴消息。
- 只有 `@盼达 /status` 在工作时段内回复动态运行状态，并优先于互动回嘴和普通消息判断。
- 单独发送 `/status` 时保持静默，不累计活跃度，也不参与能量或时间兜底判断。
- 状态命令不累计活跃度、不改变能量或冷却、不消耗回嘴机会，也不计为主动发言。
- 状态命令可展示运行阶段、能量、当前随机门槛、当前概率档位、冷却剩余时间、保护静默、匿名活跃人数与轮次、今日冒泡次数和规则版本。
- 状态命令不得公开关键词、概率阶梯、门槛范围、冷却范围、每日上限、成员身份和个人数据。

### 首次进群

- 首次进入目标群只发送一次简短欢迎语。
- 欢迎语说明机器人不是准点报时工具，但不公开能量、门槛、概率、冷却和每日上限。
- 不提供强制召唤命令；`/status` 只展示当前动态状态，不公开完整内置规则。

## 短期上下文与隐私

- 规则判断当前消息时，可以临时参考最近 10 分钟内最多 5 条真实成员文字消息。
- 群聊正文只保存在进程内存中，最长保留 30 分钟，禁止写入数据库和普通日志。
- 图片、文件、贴纸和转发内容不进入语义上下文。
- 持久化数据仅包含运行所需的派生状态：群标识、日期、能量、门槛、触发次数、冷却状态、文案编号、配置版本、幂等事件 ID 和匿名互动效果。
- 幂等事件 ID 必须设置合理的过期清理策略。
- 日志不得输出完整消息正文、飞书凭证、访问令牌或其他敏感信息。
- 不保存个人贡献分数、个人发言统计或回应者身份。

匿名互动效果允许保存：

- 文案编号与主题。
- 独立发送或回复发送。
- 发送时间。
- 表情回应总数。
- 5 分钟内是否有人回复。
- 是否触发过一次回嘴。

## 文案规范

第一版使用约 60 条经过审核的固定文案，并通过主题、发送形式和少量词槽进行随机组合。文案主题包括技术梗、牢门梗、游戏梗、摆烂梗、饭点梗和低调短句。

文案风格要求：

- 使用一到两行短句，符合高频、碎片化群聊节奏。
- 语气直接、轻松、带技术苦中作乐和互相接梗的感觉。
- 可以使用“牛马、牢门、bug、token、DLC、孩子们、致敬”等群内日常表达。
- 避免企业播报腔、鸡汤、正式汇报语气和精确进度条。
- 不复述真实项目、客户、订单、链接和业务细节。
- 不点名评价成员的能力、效率、态度或工作量。
- 不拿线上事故、客诉、裁员、薪资、疾病等严肃事项开玩笑。
- 不把加班描述为值得庆祝的行为。
- 不生成针对个人的侮辱性内容。

回嘴文案单独维护约 15～20 条，并遵守相同的安全边界。

## AI 职责与接入路线

### 第一阶段：纯规则 MVP

- 运行时不调用 AI。
- 使用短语、否定关系、风险规则和短期上下文进行分类。
- 使用固定审核文案库完成发送。
- 通过影子模式验证误判、频率和时段分布。

### 第二阶段：语义分类

- 仅将规则无法确定的候选消息交给 AI。
- AI 输出结构化分类、置信度、信号强度和风险标记。
- 最终能量、门槛、概率、冷却和发送决定仍由规则引擎负责。

### 第三阶段：现场文案

- 规则引擎确认允许发送后，AI 才能偶尔生成现场文案。
- AI 输出必须经过长度、敏感词、禁区和个人指向检查。
- AI 超时、失败或输出不合格时自动回退到固定文案库。

### 第四阶段：主题权重学习

- 仅根据匿名表情数量和短期互动结果调整文案主题权重。
- 不根据个人身份学习，不构建成员画像。

外部 AI 每次最多接收当前候选消息和两条必要上文。发送前必须删除姓名、`@`、链接和可识别编号。禁止发送图片、文件、转发内容和完整聊天历史。

## 工程架构

核心规则必须与飞书、数据库、AI 服务和系统时间解耦。建议模块边界：

```text
src/panda_bot/
├── domain.py          # 标准事件、群状态和决策结果
├── engine.py          # 纯规则状态机
├── classifier.py      # 信号、否定和风险识别
├── probability.py     # 门槛、衰减和抽签
├── copywriter.py      # 文案选择和近期去重
├── repository.py      # 状态存储接口
├── settings.py        # 外部配置加载
├── adapters/          # 飞书、SQLite 和未来 AI 适配器
└── main.py            # 应用启动与依赖组装
```

工程要求：

- 使用依赖注入提供当前时间和随机数生成器，测试中必须支持固定时钟和随机种子。
- 核心引擎不得直接执行网络请求或数据库操作。
- 飞书事件必须先完成验签、幂等去重和标准化，再交给核心引擎。
- 发送前再次执行时间、冷却、每日上限和风险状态检查。
- SQLite 写入需要事务保护，状态更新与发送结果必须具备明确的一致性策略。
- 外部配置承载业务词表、分数、概率、时段、冷却和文案，禁止在代码中硬编码具体业务数据。

## Python 编码规范

- 使用 `uv` 管理 Python、依赖和锁文件。
- 引入第三方依赖前验证其与项目 Python 版本的兼容性。
- 所有行内注释、文档字符串和 TODO 使用中文。
- 主要流程和核心处理函数必须包含中文 docstring，说明功能、每个参数的作用和返回值。
- 关键流程使用标准 `logging` 模块记录开始、结束、分支结果、外部资源加载、异常和降级；禁止使用 `print` 充当运行日志。
- 正常关键节点使用 `INFO`，异常和降级使用 `WARNING` 或 `ERROR`。
- 优先使用类型标注、不可变领域对象和小型纯函数。
- 禁止把飞书密钥、数据库凭证和真实群标识硬编码进源码。
- 配置缺失或不合法时快速失败，并提供清晰错误信息。

## 测试要求

至少覆盖以下场景：

- 重复事件、机器人消息、图片、贴纸和转发内容被正确忽略。
- 同一成员 60 秒内的连续拆句正确合并。
- 上午能量经过午休衰减后影响下午。
- `14:30` 前无法发送，`17:30` 后无法发送。
- 收尾消息优先使用能量触发流程，普通消息只能在时间兜底规则中低概率触发且不改变能量。
- 单一成员的明确完成信号在群体活跃条件满足时可以推动触发。
- “还没完成”“完成不了”“准备返工”等否定表达不增加能量。
- 日常吐槽不触发事故沉默，明确业务事故触发沉默。
- 每日最多主动发言 4 次，随机冷却位于 30～45 分钟。
- 固定时钟和随机种子可以完全复现决策结果。
- 文案近期不重复，独立发送和回复发送按配置随机切换。
- 每次主动发言最多产生一次回嘴，且回嘴遵守 `17:30` 截止时间。
- 数据库不保存消息正文和个人互动身份。
- 进程重启后不会重复处理已经消费的飞书事件。
- AI 超时、异常和不合格输出能够回退到纯规则与固定文案。

正式发言前必须运行 3～5 个工作日影子模式。影子模式只记录决策原因、匿名统计和“本应发送”的文案编号，不向群内发送消息。

## 本地运行与 Docker

- 开发阶段使用 `uv run` 在本地运行。
- 飞书事件优先通过长连接接收，不要求本地开放公网端口。
- 提供 `Dockerfile` 和 `compose.yaml` 作为统一交付方式。
- 容器使用非 root 用户运行，支持健康检查、优雅退出和异常自动重启。
- SQLite 数据目录挂载为持久化卷，重新创建容器后状态仍然存在。
- 飞书凭证通过环境变量或未提交的本地 `.env` 注入。
- 镜像中不得包含真实凭证、群聊正文或本地生产数据库。

## Git 规范

- 提交标题使用 Conventional Commits 格式，例如 `feat:`、`fix:`、`docs:`、`refactor:`、`chore:`、`test:`。
- 冒号后的标题和正文使用中文。
- 提交正文要点使用 Markdown 短横线。
- 提交信息不得添加 `Co-Authored-By: Claude` 等协作者署名。
- 保留用户已有修改，不覆盖与当前任务无关的工作区内容。
- 每次完成文件修改并通过相关验证后立即创建提交，不保留已完成的未提交改动。

任务只有在以下条件满足后才算完成：

- 关键流程具有中文 docstring、必要日志和自动化测试。
- 相关测试通过，并报告实际验证结果。
- 文档与当前实现一致，只描述最终状态。
- 不遗留真实凭证、聊天正文、临时调试文件或未说明的降级行为。
